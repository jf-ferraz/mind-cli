## Overview

Phase A establishes two foundational capabilities that every subsequent improvement depends on:

1. **State File (G1)** — Persistent, file-based tracking of conversation progress, phase outputs, and runtime metrics. Enables resume-after-interruption, conditional routing (Phase B), and quality loops (Phase C).

2. **Context Isolation Rules (G2)** — Declarative rules that specify exactly what context each persona receives at each phase. Replaces implicit moderator discipline with auditable, versioned rules.

These are **additive** changes — they don't break any existing behavior. A conversation run without the state file works exactly as today; the state file adds checkpointing on top.

### Design Principles

| Principle | Application |
|-----------|-------------|
| Files are memory | State lives in git-tracked (or gitignored ephemeral) files, not in-context |
| Minimum viable context | Each persona loads only what context rules permit |
| Orchestrator owns state | Only the moderator reads/writes the state file |
| Backward compatible | All additions are opt-in with sensible defaults |

## G1: Conversation State File

### Rationale

The moderator currently holds all workflow state in its context window:
- Which phases have been completed
- Where phase outputs are stored
- Quality scores from previous phases
- Configuration decisions made at runtime

This creates three failure modes:

1. **Context compaction** — VS Code Copilot may compact the context window during long conversations, losing phase tracking information.
2. **Session interruption** — If the user closes the session, there's no way to resume from the last completed phase. The entire conversation must restart.
3. **State invisibility** — The user can't inspect conversation progress without reading through the entire chat history.

### State File Schema

The state file is a YAML document written to `.github/state/conversation-state.yml`.

```yaml
# ═══════════════════════════════════════════════════════════════
# Conversation State — Ephemeral Session Tracker
# ═══════════════════════════════════════════════════════════════
# This file is auto-generated by the conversation-moderator agent.
# Do NOT edit manually during an active session.
# Safe to delete between sessions.
# ═══════════════════════════════════════════════════════════════

# ─── SESSION IDENTITY ──────────────────────────────────────────
session:
  id: ""                          # Unique session identifier (date-slug format)
  topic: ""                       # Short topic description (from config)
  variant: ""                     # Variant preset used (quick/deep/devils_advocate/panel/document_as_position/custom)
  config_path: ""                 # Path to the config file used
  started_at: ""                  # ISO 8601 timestamp
  updated_at: ""                  # Last state update timestamp
  status: "active"                # active | paused | completed | aborted

# ─── PERSONA ROSTER ───────────────────────────────────────────
# Resolved from config at session start. Tracks effective personas
# after diversity audit may have removed duplicates.
personas:
  configured: []                  # List of persona IDs from config
  effective: []                   # List after diversity audit (may differ)
  removed: []                     # Personas removed by diversity audit (with reason)

# ─── PHASE PROGRESS ───────────────────────────────────────────
phases:
  phase_2_opening:
    status: "not_started"         # not_started | in_progress | completed | skipped | failed
    started_at: null
    completed_at: null
    outputs: []                   # List of {persona, path, word_count, status}
    notes: ""                     # Moderator notes (e.g., "re-run needed for persona X")

  phase_2_5_diversity_audit:
    status: "not_started"
    completed_at: null
    effective_persona_count: null
    similarity_matrix: {}         # Pairwise similarity scores {pair_key: score}
    actions_taken: []             # e.g., ["consolidated architect+pragmatist", "injected devil_advocate"]

  phase_2_7_tension_extraction:
    status: "not_started"
    completed_at: null
    tension_count: null
    output_path: null

  phase_3_cross_examination:
    status: "not_started"
    round: 1                      # Current round (supports multi-round via G3)
    started_at: null
    completed_at: null
    outputs: []

  phase_4_rebuttal:
    status: "not_started"
    round: 1
    started_at: null
    completed_at: null
    outputs: []
    concession_count: null        # Total concessions across all personas

  phase_5_convergence:
    status: "not_started"
    iteration: 1                  # Evaluator-optimizer iteration (G4)
    started_at: null
    completed_at: null
    output_path: null
    quality_scores: null          # Populated after rubric scoring

  phase_5_5_gap_fill:
    status: "not_started"
    trigger: null                 # What triggered gap-fill (auto/manual/condition)
    gap_personas: []              # Gap-fill persona definitions
    completed_at: null

  phase_6_output:
    status: "not_started"
    output_path: null
    completed_at: null

# ─── RUNTIME METRICS ──────────────────────────────────────────
metrics:
  sub_agent_calls: 0              # Total sub-agent invocations
  sub_agent_budget: 25            # From termination.token_budget.max_sub_agent_calls
  phases_completed: 0
  phases_skipped: 0
  total_concessions: 0
  quality_score: null             # Overall quality rubric score (Phase 5)
  quality_dimensions: {}          # Per-dimension scores

# ─── CONDITIONAL ROUTING LOG ──────────────────────────────────
# Tracks routing decisions made by the moderator (G3).
routing_log: []
  # Each entry: {phase, condition_evaluated, result, action_taken, timestamp}

# ─── TERMINATION ──────────────────────────────────────────────
termination:
  reason: null                    # normal | early_convergence | abort | token_budget | user_request
  triggered_at: null
  details: ""
```

### State File Lifecycle

```
┌──────────────────────────────────────────────────────────┐
│                    STATE FILE LIFECYCLE                    │
│                                                            │
│  1. SESSION START                                          │
│     Moderator reads config → creates state file            │
│     Sets session.id, topic, variant, started_at            │
│     Resolves persona roster                                │
│                                                            │
│  2. PHASE ENTRY                                            │
│     Before spawning sub-agents:                            │
│     → Read state file                                      │
│     → Set phase.status = "in_progress"                     │
│     → Set phase.started_at                                 │
│     → Write state file                                     │
│                                                            │
│  3. SUB-AGENT COMPLETION                                   │
│     After each sub-agent returns:                          │
│     → Read state file                                      │
│     → Append to phase.outputs                              │
│     → Increment metrics.sub_agent_calls                    │
│     → Write state file                                     │
│                                                            │
│  4. PHASE COMPLETION                                       │
│     After all sub-agents for a phase complete:             │
│     → Set phase.status = "completed"                       │
│     → Set phase.completed_at                               │
│     → Increment metrics.phases_completed                   │
│     → Evaluate routing conditions (G3)                     │
│     → Write state file                                     │
│                                                            │
│  5. SESSION END                                            │
│     → Set session.status = "completed" | "aborted"         │
│     → Set termination.reason                               │
│     → Final state file write                               │
│     → State file preserved for audit (not auto-deleted)    │
│                                                            │
└──────────────────────────────────────────────────────────┘
```

### Phase Output Storage

Each sub-agent's output is written to a file in `.github/state/` rather than held only in the moderator's context.

```
.github/state/
├── conversation-state.yml                 # Session state tracker
├── topic-brief.md                         # The topic brief (Phase 1 input)
├── phase-2/
│   ├── architect-position.md              # Architect's opening position
│   ├── pragmatist-position.md             # Pragmatist's opening position
│   └── critic-position.md                 # Critic's opening position
├── phase-2.5/
│   └── diversity-audit.md                 # Diversity audit results
├── phase-2.7/
│   └── tension-matrix.md                  # Implicit tension extraction
├── phase-3/
│   ├── architect-challenges.md            # Architect's challenges to others
│   ├── pragmatist-challenges.md
│   └── critic-challenges.md
├── phase-4/
│   ├── architect-revised-position.md      # Architect's rebuttal/refinement
│   ├── pragmatist-revised-position.md
│   └── critic-revised-position.md
├── phase-5/
│   ├── convergence-analysis.md            # Moderator's convergence synthesis
│   └── quality-rubric-scores.yml          # Rubric dimension scores

└── phase-6/
```

### Resume Protocol

When the moderator starts and detects an existing state file:

```
1. READ .github/state/conversation-state.yml
2. IF session.status == "completed" OR session.status == "aborted":
   → Ask user: "A previous session exists ({session.id}). Start fresh or review?"
3. IF session.status == "active" OR session.status == "paused":
   → Identify current_phase and its status
   → IF phase.status == "in_progress":
     → Check which sub-agent outputs exist vs. expected
     → Report: "Session {id} interrupted during {phase}. {N}/{M} outputs complete."
     → Resume from the first missing output
   → IF phase.status == "completed":
     → Advance to next phase per routing logic
4. Validate all referenced output files exist on disk
5. Resume execution
```

### Gitignore Configuration

Add to project `.gitignore`:

```gitignore
# Conversation analysis ephemeral state (session-specific, not committed)
.github/state/
```

The state directory is **ephemeral** — it tracks in-flight conversation sessions. Completed analyses get their final output written to `docs/analysis/` (which IS committed). The state files are audit artifacts useful during a session but not part of the permanent project record.

> **Exception:** If the user wants to commit session state for reproducibility, they can remove the gitignore entry. The state files are valid YAML/Markdown and version-control friendly.

### Edge Cases

| Scenario | Handling |
|----------|----------|
| **State file missing mid-session** | Moderator detects missing file, recreates from current context if possible, logs warning |
| **State file corrupted (invalid YAML)** | Moderator reports corruption, asks user: continue without state or abort |
| **Concurrent sessions** | Not supported — session.id uniqueness prevents collision, but only one active session at a time |
| **Output file missing** | Moderator re-runs the specific sub-agent call that produced the missing output |
| **Disk full** | Moderator falls back to in-context state tracking (current behavior) with warning |

## G2: Explicit Context Isolation Rules

### Rationale

Context isolation is the workflow's most critical correctness property. If Persona A's Phase 2 output leaks into Persona B's Phase 2 prompt, the entire dialectic collapses — positions are no longer independent, and the adversarial value of the conversation is undermined.

Currently, isolation is enforced by:
1. Moderator instructions saying "don't leak context"
2. The structure of sub-agent prompt templates
3. Moderator discipline during prompt construction

This is a **process-based** guarantee — it works when the moderator follows instructions perfectly, but provides no structural enforcement or auditability.

### Context Rule Schema

Add a `context_rules` section to `conversation-analysis.yml`:

```yaml
# ─── CONTEXT ISOLATION RULES ──────────────────────────────────
# Declarative rules controlling what each persona sees per phase.
# The moderator MUST read these rules before constructing any
# sub-agent prompt and include ONLY the specified context.
#
# Context tokens:
#   topic_brief         → The Phase 1 topic brief document
#   own_position        → This persona's Phase 2 position paper
#   other_positions     → All OTHER personas' Phase 2 position papers
#   all_positions       → All Phase 2 position papers (including own)
#   own_challenges      → Challenges directed AT this persona (Phase 3)
#   other_challenges    → Challenges from OTHER personas (Phase 3)
#   all_challenges      → All Phase 3 challenge documents
#   challenges_received → Alias for own_challenges
#   own_rebuttal        → This persona's Phase 4 revised position
#   all_rebuttals       → All Phase 4 revised positions
#   tension_matrix      → Phase 2.7 tension extraction output
#   diversity_audit     → Phase 2.5 audit results
#   all                 → Everything from all prior phases
#   persona_definition  → The persona's own config (name, perspective, priorities, bias)
#
context_rules:

  # Phase 2: Complete isolation — independent position generation
  phase_2_opening:
    receives:
      - topic_brief
      - persona_definition
    blocked:
      - other_positions
      - all_challenges
      - all_rebuttals
    rationale: "Positions must be generated independently to prevent confirmation bias"

  # Phase 2.5: Moderator-only — personas don't participate
  phase_2_5_diversity_audit:
    actor: moderator
    receives:
      - all_positions
    rationale: "Moderator compares positions for redundancy"

  # Phase 2.7: Moderator-only — extract implicit tensions
  phase_2_7_tension_extraction:
    actor: moderator
    receives:
      - all_positions
      - diversity_audit
    rationale: "Moderator builds tension matrix from overlapping evaluations"

  # Phase 3: See others, not self — force genuine engagement with foreign ideas
  phase_3_cross_examination:
    receives:
      - topic_brief
      - persona_definition
      - other_positions
    blocked:
      - own_position
      - all_challenges
      - all_rebuttals
    rationale: "Prevents defensive posture — persona engages fresh with others' ideas"

  # Phase 4: See own + challenges received — targeted defense/concession
  phase_4_rebuttal:
    receives:
      - persona_definition
      - own_position
      - challenges_received
    blocked:
      - other_positions
      - other_challenges
      - all_rebuttals
    rationale: "Focused rebuttal — persona addresses only challenges to their position"

  # Phase 5: Moderator sees everything — full dialectic synthesis
  phase_5_convergence:
    actor: moderator
    receives:
      - all
    rationale: "Moderator needs full arc for accurate synthesis"

  # Phase 5.5: Gap-fill personas get targeted context
  phase_5_5_gap_fill:
    receives:
      - topic_brief
      - persona_definition
      - convergence_analysis    # So gap-fill personas know what gaps to address
    blocked:
      - all_positions           # Fresh perspective, not influenced by existing positions
      - all_challenges
      - all_rebuttals
    rationale: "Gap-fill personas must bring genuinely new perspectives"
```

### Context Resolution Algorithm

When the moderator constructs a sub-agent prompt, it follows this algorithm:

```
FUNCTION resolve_context(persona_id, phase_id, context_rules, state):
  1. Look up rules = context_rules[phase_id]
  2. IF rules.actor == "moderator":
     → This phase is moderator-only, no sub-agent prompt needed
     → RETURN

  3. context_bundle = {}

  4. FOR EACH token IN rules.receives:
     → Resolve token to actual content:
       - "topic_brief"         → read state topic-brief.md
       - "persona_definition"  → read persona config from YAML
       - "own_position"        → read phase-2/{persona_id}-position.md
       - "other_positions"     → read all phase-2/*.md EXCEPT {persona_id}
       - "challenges_received" → grep phase-3/*.md for challenges targeting {persona_id}
       - "all"                 → read all phase output files
     → Add resolved content to context_bundle

  5. VALIDATE: For EACH token IN rules.blocked:
     → Verify that context_bundle does NOT contain content matching the blocked token
     → If violation detected: LOG WARNING, remove offending content

  6. RETURN context_bundle
```

### Isolation Audit Trail

The moderator logs context delivery in the state file for every sub-agent call:

```yaml
# Addition to phase output entries in conversation-state.yml
outputs:
  - persona: architect
    path: ".github/state/phase-3/architect-challenges.md"
    status: "completed"
    context_delivered:             # Audit: what context was actually provided
      - token: "topic_brief"
        source: ".github/state/topic-brief.md"
      - token: "other_positions"
        sources:
          - ".github/state/phase-2/pragmatist-position.md"
          - ".github/state/phase-2/critic-position.md"
    context_blocked:              # Audit: what was explicitly excluded
      - token: "own_position"
        would_have_been: ".github/state/phase-2/architect-position.md"
```

This creates a verifiable audit trail. After a conversation completes, anyone can inspect the state file to confirm that isolation was maintained.

### Custom Context Patterns

The default rules handle standard variants. For custom workflows, users can define novel patterns:

```yaml
# Example: "Partial visibility" — persona A can see persona B but not C
context_rules:
  phase_3_cross_examination:
    custom_per_persona:
      architect:
        receives: [topic_brief, persona_definition]
        receives_from: [pragmatist]     # Only sees pragmatist's position
        blocked_from: [critic]          # Does NOT see critic's position
      pragmatist:
        receives: [topic_brief, persona_definition]
        receives_from: [critic]
        blocked_from: [architect]
      critic:
        receives: [topic_brief, persona_definition]
        receives_from: [architect]
        blocked_from: [pragmatist]
```

This enables **ring topology** (each persona only sees the next), **star topology** (one persona sees all, others see only that one), or any custom visibility graph.

### Validation Checklist

After implementing G2, verify:

- [ ] Default context rules match the existing isolation strategy documented in the workflow
- [ ] Moderator reads `context_rules` before constructing ANY sub-agent prompt
- [ ] Moderator logs `context_delivered` and `context_blocked` for every sub-agent call
- [ ] Custom `per_persona` rules override default rules correctly
- [ ] Blocked content is never present in delivered context (negative test)
- [ ] Phase 2 positions show no evidence of cross-contamination (content analysis)

## Implementation Sequence

### Step 1: Create state directory and gitignore

```bash
mkdir -p .github/state
echo ".github/state/" >> .gitignore
```

### Step 2: Add state management to YAML config

Add to `conversation-analysis.yml`:

```yaml
# ─── STATE MANAGEMENT ─────────────────────────────────────────
state:
  enabled: true
  path: ".github/state"
  auto_cleanup: false             # If true, delete state dir after Phase 6 completes
  resume_on_restart: true         # If true, moderator checks for existing state on start
```

### Step 3: Add context rules to YAML config

Add the full `context_rules` section from above to `conversation-analysis.yml`.

### Step 4: Update moderator agent instructions

Add to `conversation-moderator.md`:

1. **State management block** — Instructions for reading/writing state file at phase boundaries
2. **Context rule enforcement block** — Instructions for reading `context_rules` and constructing prompts accordingly
3. **Resume detection block** — Instructions for detecting and resuming interrupted sessions
4. **Audit logging block** — Instructions for recording `context_delivered` and `context_blocked`

### Step 5: Update workflow documentation

Add sections to `conversation-analysis-workflow.md`:

1. **State File** subsection under Architecture
2. **Context Isolation Rules** subsection under Architecture
3. **Resume Protocol** subsection under Execution Patterns
4. **Isolation Audit** subsection under Quality Assurance

### Step 6: Validate

Run a test conversation and verify:
- State file is created and updated at each phase boundary
- Phase outputs are written to `.github/state/phase-*/`
- Context audit trail shows correct isolation
- Interrupting and resuming a session works correctly

## Validation Criteria

### G1 Pass Criteria

| Criterion | Test |
|-----------|------|
| State file created on session start | Check `.github/state/conversation-state.yml` exists after first phase |
| Phase status transitions correctly | Status goes `not_started` → `in_progress` → `completed` |
| Output files persisted | Each sub-agent output exists as a file in `.github/state/phase-N/` |
| Sub-agent call counter increments | `metrics.sub_agent_calls` matches actual call count |
| Resume works after interruption | Stop mid-Phase 3, restart, verify Phase 3 resumes where it left off |
| Session completion recorded | `session.status == "completed"` after Phase 6 |

### G2 Pass Criteria

| Criterion | Test |
|-----------|------|
| Rules read before prompt construction | Moderator output references context rules |
| Phase 2 isolation holds | No position paper references another persona's arguments |
| Phase 3 isolation holds | No challenge document references the persona's own position |
| Audit trail complete | Every sub-agent call has `context_delivered` and `context_blocked` entries |
| Custom rules work | Ring topology produces expected visibility pattern |
| Blocked content absent | Search phase outputs for content that should have been blocked |
