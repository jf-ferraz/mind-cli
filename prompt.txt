## Context

We are building **mind-cli** — a CLI/TUI interface for our existing **Mind Agent Framework** (a Claude Code-based agent system with 7 core agents, 4-zone documentation model, and structured software development workflows).

## Approach A: Rust Backend Orchestrator (`archive/`)

A **state management engine** inspired by NixOS's flake.lock pattern, with three pillars:

- **Governance Layer** (`mind.toml`): Declarative TOML manifest defining project identity, document registry, dependency graph, workflows, and governance policies.
- **Reconciliation Layer** (`mind.lock`): JSON lock file with pre-computed hashes, transitive staleness detection, and completeness metrics. Change in one artifact propagates downstream automatically.
- **Execution Layer**: 7 agents orchestrated adaptively with probabilistic (micro-gates) and deterministic (build/lint/test) quality gates.

**Tech stack**: Bash dispatcher → Python 3.11+ MVP (zero external deps, stdlib only) → Rust rewrite in Phase 2 (clap, serde, tokio). Target: sub-5ms startup, single static binary.

**Key strength**: The reconciliation engine — reactive dependency graph, hash-based staleness tracking, and a CLI that acts purely as a state engine while agents remain the decision engine. All data contracts are JSON-schema-versioned.

**What was completed**: Full architectural spec, MVP implementation plan (7-step delivery, ~800 LOC), data contracts, performance targets, adoption levels (L0-L3), risk mitigations. **No implementation code was written.**

## Approach B: Interface-Driven Go CLI/TUI (`docs/blueprints/`)

A **project intelligence layer** — purely deterministic, no AI API calls — that bridges the gap between human developers and AI agent workflows through 4 integration models:

- **Model A (Pre-Flight)**: `mind preflight "request"` — classifies request, validates docs (17 checks), creates iteration/branch, assembles context, generates prompt. Low complexity, high value.
- **Model B (MCP Server)**: `mind serve` — exposes 16 structured tools to any MCP-compatible agent (status, validation, iteration management, state read/write, quality tracking). Medium complexity, very high value. **This is the inflection point** — once built, Models C/D become much simpler.
- **Model C (Sidecar/Watch)**: `mind watch` — real-time filesystem monitoring with live validation. Medium complexity, high value.
- **Model D (Full Orchestration)**: `mind run "request"` — drives entire pipeline headlessly via `claude` CLI, dispatches agents, runs gates deterministically, manages retries. High complexity, very high value. ~20-30% token savings vs orchestrator agent approach.

**Tech stack**: Go 1.23+, Cobra (CLI), Bubble Tea + Lip Gloss (TUI), fsnotify (watch), GoReleaser (cross-platform distribution). 4-layer architecture: Presentation → Service → Domain → Infrastructure. Repository pattern for testability.

**Key design decision**: "No AI in CLI" — the CLI is strictly deterministic. AI workflows stay in Claude Code/agents. The CLI is the **companion tool** handling everything around AI workflows (doc management, validation, state tracking, scaffolding).

**What was completed**: 3 detailed blueprints (CLI/TUI, AI Workflow Bridge, Software Architecture), project brief, domain model, command tree (~13 commands), implementation phases (5 core + 4 integration models). **No implementation code was written.**

## Shared Foundation (Both Approaches)

- **4-zone documentation model**: spec (stable), state (volatile), iterations (immutable history), knowledge (reference)
- **7 core agents**: Orchestrator, Analyst, Architect, Developer, Tester, Reviewer, Discovery + Conversation Moderator
- **Quality gates**: Probabilistic (analyst, architect review) + deterministic (build/lint/test) + evidence-based (reviewer)
- **`mind.toml` manifest**: Project identity, document registry, governance policies
- **Iteration-based workflow**: Per-change tracking with typed iterations (NEW_PROJECT, BUG_FIX, ENHANCEMENT, REFACTOR, COMPLEX_NEW)

## The Decision Point

Both approaches are fully designed but unimplemented. Key tensions:

| Dimension | Rust/Reconciliation (A) | Go/Interface-Driven (B) |
|-----------|------------------------|------------------------|
| **Core value** | Reactive state engine with transitive staleness | Project intelligence + 4-model AI bridge |
| **Complexity** | NixOS-inspired lock/reconciliation is sophisticated | Simpler core, complexity in integration models |
| **AI integration** | Agents read CLI JSON output | MCP server + orchestration via `claude` CLI |
| **Performance** | Sub-5ms (Rust), <1s lock sync | <50ms startup (Go), <200ms status |
| **Distribution** | Single binary (Rust Phase 2) | Single binary (Go from day 1) |
| **TUI** | Not designed | Full Bubble Tea dashboard (5 tabs) |
| **Lock file** | Central innovation (hash + staleness graph) | Not present |
| **MCP server** | Deferred to Phase 3 | Core integration model (Model B) |

## Constraints & Goals

- **No deadline pressure** — we can invest in a harder path if it delivers more long-term value.
- The end goal is to elevate the Mind Framework into a **robust, scalable engine** — not just a thin wrapper.
- The chosen path must balance: implementation complexity, architectural scalability, developer experience, and the value delivered to end users.
- Current framework runs on Claude Code with markdown agents and bash scripts — the CLI must enhance, not replace, this foundation.
